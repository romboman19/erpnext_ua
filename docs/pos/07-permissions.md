# 07 — Ролі та матриця дозволів

## 1. Ролі (fixtures)

| Роль | Призначення |
|---|---|
| **POS Касир** | базові операції продажу на своїй касі |
| **POS Старший касир** | + повернення за межами дня, перегляд інших складів, прийом інкасацій |
| **POS Менеджер** | + approvals (коди), знижки, мінусові продажі, розбіжності, звіти по всіх касах |
| **POS Адміністратор** | конфігурація кас, допуски, правила ФОП, force-close змін, ручні виправлення станів |
| **ПРРО Оператор** | фіскальні зміни/Z-звіти/службові операції (зазвичай = касир, але роль окрема) |
| **Kiosk-user** (технічний) | логін станції; НЕ має прав на дії — все через POS-сесію працівника |

Ролі складаються: касир може мати ПРРО Оператора. Чутливі дії вимагають **і** роль,
**і** активну POS-сесію відповідного Employee, **і** (де вказано) одноразовий код менеджера (M).

## 3. Реалізація

- DocType-permissions: стандартні role permissions Frappe (read/write/submit/cancel) —
  але submit/cancel фінансових документів доступні лише системному коду (`ignore_permissions`
  всередині API після власних перевірок), у ролей — тільки read; це виключає обхід POS API через Desk.
- Дія-рівень: `services/permissions.py` — єдина точка `check(action, session, context)`;
  усі whitelisted-методи викликають її першою.
- Manager approval: POS Manager Authorization (03) — код одноразовий, TTL 5 хв, прив'язаний
  до action_type + scope; перевіряється сервером, спалюється атомарно.
- Employee-рівень: Employee Cash Desk Access перевіряється на кожен виклик (роль ролью,
  а допуск до конкретної каси — окремо).
- Другий фактор (майбутнє): поле `ua_pos_pin_hash` вже в моделі; вмикається политикою профілю.
- Кожна клітинка «M/⚙/журнал» відображається у POS Event Log з контекстом (хто/де/що/чому).
