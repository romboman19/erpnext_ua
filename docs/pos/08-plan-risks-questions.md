# 08 — План реалізації, ризики, blocking questions, міграція

## 1. План по етапах

Кожен етап закінчується працюючим інкрементом на тестовому сайті + автотестами + короткою
інструкцією. Відносна вага — у «тижнях розробки» одного розробника (орієнтир, не комітмент).

| Етап | Зміст | Вага | Залежності |
|---|---|---|---|
| **0. Фіксація рішень** | закриття blocking questions 1–5, фінал моделі даних, тестовий стенд (окремий site), тестовий ФОП з ключами ДПС (cabinet.tax.gov.ua:9443) | 1 | — |
| **1. Каркас і каса** | модуль ua_pos, DocTypes конфігурації, допуск по штрихкоду, POS-сесії, Operational Shift + Denomination Count + Cash Movement, інкасація/витрати/передачі, журнал подій, касовий звіт | 3–4 | 0 |
| **2. Продаж (нефіскальний)** | POS Order + UI сторінка (скан, кошик, клієнт, ціни/акції/лояльність через сервер), checkout-saga для готівки, SI (is_pos, update_stock), lookup-токен, серії/партії/FEFO/мінусовий продаж + службовий чек, друк server-side (черга, нефіскальний чек, гарантійний талон), повернення нефіскальні | 5–6 | 1 |
| **3. Фіскалізація** | FiscalAdapter поверх ua_fiscal, розширення PRRO Shift/Receipt, фіскальні зміни по ФОП (відкриття за першим продажем — див. питання №8), X/Z, службові вн./вид. (зв'язані з Movement), фіскальний продаж/повернення, ретраї/QueryState, офлайн-сесія за flag, e2e на тестовому API ДПС | 4–5 | 2 |
| **4. Термінал і змішана оплата** | доробки pb-pos-gateway (status/void/idempotency), TerminalAdapter, Payment Attempt/Terminal Transaction, unknown-процедура, змішана оплата (фіскальна, UAH), звірка терміналів | 3–4 | 2 (частк. ‖ 3) |
| **5. МультиФОП-контроль** | Fiscal Routing Rule + движок + пояснення, політики ФОП, спліт кошика, ліміти (джерело: ПРРО+IBAN+коригування), dashboard, контрольні звіти по ФОП | 3 | 3 |
| **6. Рахунки і передоплата** | Sales Order-рахунок, передоплати (PE), резерв (Stock Reservation), видача без подвійного списання, IBAN-підтвердження (банк-імпорт з ukrainian_integrations) | 3 | 2 |
| **7. Відкладене** (за окремими flag) | подарункові сертифікати, кредит, ідентифікація клієнта (SMS/дзвінок/Telegram), готівка USD/EUR (після юр. висновку), hardware-agent (якщо потрібен) | 3+ | 5,6 |
| **8. Hardening і приймання** | повний прогін 36 acceptance-сценаріїв, звірка ERP/ПРРО/термінал/каса, навантаження 3 каси, документація касира/адміністратора, навчання, пілот на 1 касі → 3 каси | 2–3 | усі |

Паралелізація: 3 і 4 частково паралельні після 2; 5 — після базової фіскалізації.
**Перший продуктивний реліз** можливий після етапу 3 (одна каса, один ФОП, готівка + фіскалізація).

## 2. Ризики

| # | Ризик | Ймовірність/Вплив | Мітигація |
|---|---|---|---|
| R1 | Протокол ДПС: нюанси CAdES-T (TSP) для онлайн-чеків, поведінка тестового vs прод серверів | В/В | етап 0: e2e-спайк на тестовому API з реальними тестовими ключами; prro_docs дзеркало вже на сервері; ppolib як референс |
| R2 | Legacy-протокол терміналів ПБ не дає надійного status-by-id | С/В | доробка pb-pos-gateway (журнал ECR/ServiceMessage); якщо неможливо — процедура ручної звірки з чеком термінала + settlement наприкінці дня |
| R3 | Модель «одна Company» не переживе юридичну перевірку (роздільний облік ФОП) | С/В | blocking question №1; архітектура фази N готова; виміри пишуться з дня 1 |
| R4 | Продуктивність скан→кошик (вимога «мінімальна затримка») при серверній валідації | С/С | локальний кеш довідника товарів (readonly) + серверна валідація асинхронно до моменту оплати; заміри на етапі 2 |
| R5 | Офлайн-ланцюг PREVDOCHASH зламається (розрив послідовності) | Н/В | єдина точка генерації номерів (lock), ланцюг у PRRO Offline Session, автотест ланцюга |
| R6 | Оновлення ERPNext ламає custom fields/hooks | Н/С | тільки дозволені механізми розширення; CI: `bench migrate` + smoke на staging перед прод; кастомний образ версіюється |
| R7 | Касири обходять POS через Desk UI | С/С | kiosk-user без прав; submit фінансових доків лише системним кодом; permissions review на етапі 8 |
| R8 | Друк ESC/POS: кирилиця/кодові сторінки конкретних принтерів | С/Н | етап 0/2: тест на реальному принтері; шаблони з явною кодовою сторінкою |
| R9 | Розповзання скоупу (сертифікати, кредит, валюта) до запуску ядра | В/С | feature flags + фази 6–7 жорстко після ядра |
| R10 | Один розробник + прод на живому магазині | В/В | пілот на 1 касі, паралельна робота зі старою системою на перехідний період, повний бекап перед кожним релізом (вже практика: bench backup) |

## 3. Blocking questions — РІШЕННЯ ЗАФІКСОВАНІ (інтерв'ю з власником, 2026-07-11)

| # | Питання | ✅ Рішення |
|---|---|---|
| 1 | Модель ФОП | **Одна управлінська Company + ФОП як вимір** (фаза 1); Company-на-ФОП — майбутня фаза зі спроєктованою міграцією |
| 2 | Обладнання друку | Чеки — **Xprinter XP-80T**, офісний — **HP LaserJet M135w**, етикетки — **XP-450B**; друк уже мережевий («через інтернет») ⇒ server-side друк підтверджено, hardware-agent не потрібен. Перевірити на етапі 0: інтерфейс XP-80T (LAN/USB) і кодову сторінку кирилиці |
| 3 | Готівка USD/EUR | **Валюта приймається фізично, але рахується як гривня** за внутрішнім курсом дня (задає адмін/менеджер, довідково поруч НБУ). Фіскально — готівка UAH на суму еквівалента. POS сам рахує еквівалент і решту (в UAH), щоб продавець не помилявся. Фізичні USD/EUR у скриньці обліковуються по валютах (перерахунки/інкасації) |
| 4 | Змішане часткове повернення | **Вибір касира в межах лімітів** по кожному способу (≤ сплаченого мінус уже повернуте) |
| 5 | Фіскалізація передоплати | Залежить від способу оплати: **картка — фіскалізувати одразу; готівка — на вибір касира (якщо так — одразу); IBAN — при підтвердженні надходження**, з питанням «фіскалізувати?» у момент відмітки оплати. ФОП авансу = ФОП видачі |
| 6 | Подарункові сертифікати | **Не фіскалізуються** (ФОП на ЄП формально не може їх використовувати). Одноразові, без видачі решти, зі строком дії. Обов'язковий облік бланків: наявність, дата продажу, номінали, терміни. Підхід обліку (товар / гроші / відкладена знижка) — порівняти всі три на етапі 7, можливо реалізувати конфігуровано |
| 7 | «Кредит» | Це **банківська розстрочка (оплата частинами)** через ТГ-бот/додаток/термінал банку. Пряма інтеграція не потрібна — окремий Mode of Payment «Оплата частинами (банк)» з ручним підтвердженням надходження. Власний кредитний workflow зі скоупу ВИЛУЧЕНО |
| 8 | Відкриття фіскальних змін | **При першому продажу по ФОП** (авто) + ручне відкриття через фіскальне меню |
| 9 | Видача за рахунком | SI(update_stock) проти Sales Order; **часткова видача дозволена**, повторна видача тих самих кількостей блокується |
| 10 | МультиФОП-кошик | **split_prompt**: система показує спліт, касир підтверджує |
| 11 | Обмін | Підтверджено: лише нефіскальні; пара «повернення + новий продаж» зі взаємними посиланнями, без окремого документа |
| 12 | Курс валют | **Внутрішній курс дня, задає адмін/менеджер** (DocType POS Currency Rate), НБУ — довідково; решта — завжди UAH |
| 13 | Ідентифікація клієнта | Перший канал — **SMS-код через TurboSMS** (інтеграція вже є) |
| 14 | Ретенція сирих логів ДПС/термінала | **3 роки** (1095 днів), далі автоархівація; фіскальні номери/суми в документах — назавжди |
| 15 | Паралельні кошики | **До 5 відкладених** (Held) із перемиканням вкладками |

### Уточнення (2-й тур інтерв'ю, 2026-07-12)

| Тема | ✅ Рішення |
|---|---|
| Решта з валюти | Завжди **гривнею** (валюту назад не видаємо) |
| Фіскалізація авансу готівкою | Вирішує **касир** (питання в момент передоплати), не автоправило |
| Гарантія | **Строк гарантії прописаний до товару** (поле `ua_warranty_months` на Item / Item Group); талон бере строк звідти |
| Джерело доходу ФОП для лімітів | **Виключно Z-звіти ПРРО** (+ ручні коригування FOP Limit Adjustment). IBAN/GL до бази ліміту не входять — знімає відкриту залежність рішення №1 щодо джерела |
| Принтери (повний парк, усі мережеві) | Офісний **HP LaserJet M135w** (WiFi); чековий **Xprinter XP-80T** (LAN); етикетки — **2× XP-450B** (один цінники, один ТТН); **Zebra** (цінники, зараз не працює, на продаж — не інтегруємо). Друк цінників/ТТН — у скоупі POS Printer (тип label) |

**Наслідок для лімітів ФОП:** розрахунок використаної суми = Σ `sales_total − refunds_total`
по Z-звітах (PRRO Shift) згрупованих по `fop_profile` + FOP Limit Adjustment. Джерело єдине
й аудитоване (Z-звіт = офіційний фіскальний підсумок зміни). Це спрощує `income_monitor` і
робить dashboard лімітів однозначним.

**Усі blocking questions закрито.** Відкритих залежностей для старту етапів 0–3 немає.

## 4. Стратегія міграції та оновлень

**Розгортання:**
- нові DocType/поля — тільки через міграції апки (`patches.txt` + fixtures); жодних ручних змін на прод;
- порядок релізу: staging site (окремий site в тому ж bench або локальний bench) → прогін тестів →
  `bench backup --with-files` → перезбирання образу `erpnext-huntervua:vX` (існуючий Containerfile-конвеєр) →
  `docker compose up -d --force-recreate` → `bench migrate` → smoke-чекліст (відомі граблі: apps.txt
  після recreate, порожній patches.txt для esbuild — уже задокументовані в пам'яті проєкту);
- GitHub remote для erpnext_ua — налаштувати до етапу 1 (зараз відсутній!), CI: lint + unit-тести.

**Оновлення ERPNext:**
- custom app не торкається ядра ⇒ мінорні апдейти безпечні; перед мажорними — прогін тестового
  сайту з новою версією і повний прогін acceptance-набору;
- контрольний список сумісності: custom fields (fieldname-колізії), hooks doc_events, приватні API
  ERPNext не використовуємо (тільки документовані контролери SI/PE/Stock).

**Міграція даних при переході на Company-на-ФОП (фаза N):**
1. створити Company для кожного FOP Profile; 2. розкласти залишки складів по ФОП (Stock Reconciliation
   у нових компаніях за правилами закріплення); 3. нові SI пишуться в компанію ФОП; історичні документи
   не переносяться (вимір FOP Profile уже містить аналітику); 4. паралельний період: звіти об'єднують
   «стара Company + нові» по виміру.

**Перехід з поточної системи обліку (Торгсофт):**
- імпорт: номенклатура + штрихкоди + залишки (Stock Reconciliation) + клієнти + відкриті зобов'язання;
- зрізи: залишки на дату переходу друкуються та підписуються (акт), каса перераховується покупюрно
  як opening_float першої зміни;
- період паралельної роботи: 1 каса на новій системі, 2 на старій → повний перехід після тижня без блокерів.
