# 09 — План тестування та acceptance-сценарії

## 1. Рівні тестування

| Рівень | Інструмент | Покриття |
|---|---|---|
| Unit (Python) | `frappe.tests` (`bench run-tests --app erpnext_ua`) | роутинг ФОП, формула очікуваної готівки, ліміти, валідації серій/партій/FEFO, стани saga, ідемпотентність |
| Unit (XML/протокол) | pytest-стиль на xml_builder | check01 проти XSD (дзеркало prro_docs), офлайн-ланцюг PREVDOCHASH, CRC32-номери |
| Інтеграційні з ДПС | тестовий API `cabinet.tax.gov.ua:9443`, тестові ключі | відкриття/закриття зміни, чек, повернення, службові, офлайн-пакет, QueryState-відновлення |
| Інтеграційні з терміналом | mock-gateway (той самий контракт) + реальний термінал на пілоті | confirmed/declined/timeout/unknown/void/status |
| E2E UI | Playwright проти тестового site | сценарії нижче, керування клавіатурою, емуляція сканера (keystroke burst + Enter) |
| Ручні | чекліст пілота | друк (кирилиця, купюри, талони), реальний ECR, реальні ключі |

Обов'язкові стенди: **test site** (mock-адаптери), **staging** (тестовий API ДПС + mock термінал),
**пілот** (прод, 1 каса). Фікстури: демо-каталог з партійними/серійними товарами, 2 тестові ФОП,
2 каси.

## 3. Критерії готовності релізу (gate етапу 8)

1. Всі 36 сценаріїв зелені на staging (авто) + пройдені вручну на пілоті (реальні ключі/термінал/принтер).
2. Жодного стану saga без автоматичного або документованого ручного виходу.
3. Звірка (36) сходиться на 3 днях пілотної роботи поспіль.
4. Довідка касира (1 сторінка гарячих клавіш) і керівництво адміністратора написані.
5. Бекап/відновлення відпрацьовані: відновлення staging з прод-бекапа + прогін smoke.
